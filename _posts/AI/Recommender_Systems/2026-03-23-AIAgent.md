## 1. Introduction
최근 LLM이 발전함에 따라 LLM 기반 에이전트들이 자율적인 상호작용과 의사결정 task에서 괜찮은 성능을 보이고 있다. (집단/개인 수준 both) <br>
그러나 <span style="color: blue">기존 연구</span>들은 주로 LLM의 의미적 지식을 활용해서 <span style="color: blue"> 인간의 대화룰 시뮬레이션하는데 초점</span>을 두고 있다. <br>
하지만 <span style="color: blue">실제 현실에서 인간은 대화 뿐만아니라 추천시스템에서의 user-item interaction처럼 비언어적 요소로도 본인의 선호도를 암묵적으로 나타낸다.</span> (이런 요소들은 개인화된 사용자 모델링을 가능하게 한다.) <br>
이런 행동을 시뮬레이션 하기 위해 <span style="color: blue"> 일부 연구는 상호작용 기록을 자연어로 변환하여 LLM에 입력하기도</span> 하는데, 이러한 방식은 상호작용의 근본적인 행동패턴을 포착함에 있어 한계가 있다. <br> 
∵ 범용 언어 모델링과 개인화된 행동 모델링 사이에 간극O <br>
예 : 금요일에 기저귀를 산사람이 맥주도 함께 사는 경향이 있다고 했을 때 CF 모델에서는 효과적으로 이러한 경향이 포착될 수 있으나 LLM은 두 아이템이 의미적 유사성은 떨어지기 때문에 혼동할 수 있다. <br><br>

그래서 LLM 기반의 시뮬레이션에서 인간의 행동을 더 잘 표현하는 방법을 연구할 필요성이 있다. <br>
-> ∴ 추천시스템을 활용하여 LLM 기반 에이전트의 user-item 상호작용 시뮬레이션을 하는 연구가 과거에도 이뤄어졌었다. <br> 과거 interaction 기반으로 LLM을 직접 프롬프팅하는 방식 이외에도 다양한 전략으로 추천했던 연구들이 기존에 있었는데,  <span style="background-color: #fff3cd">기존 연구들은 주로 user behavior modeling에 중점을 두고 있다. </span> <br>
<br>
그러나 <span style="background-color: #fff3cd">u본 논문 (이하 AgentCF)은 user agent와 item agent를 모두 시뮬레이션하고 (= 둘다 에이전트로 모델링하는데, 둘다 메모리 모듈을 가지게 된다. 추후 추가 설명 O) 전통적인 CF의 최적화과정을 모방하고자하는 것</span>이 핵심 아이디어이다. <br>
<br>
이렇게 했을 때 장점 : <br>
1. 단순히 <span style="color: blue">상호작용을 텍스트로 변환하지 않아도 되고, 사용자와 아이템 간 자율적인 상호작용을 통해 user와 item 관계 모델링 가능</span><br>
2. 사용자 행동을 개별적으로 시뮬레이션하는 것이 아니라 user와 item 에이전트 간 interaction을 통해 <span style="color: blue">양방향 관계를 공동으로 최적화</span>할 수 있다. <br>
∴ Reflexion이나 Self refine 같은 기존 연구들은 self-learning 방식(에이전트가 자기 스스로를 탐색하여 최적화하는 방식 +task focused / user focused)이나, AgentCF는 CF의 특성을 강조하여 user와 item이 함께 맞춰나가며 학습함에 차별성이 있다. <br>
<br><br>
즉,<br>
각 단계에서 (1)사용자와 아이템 에이전트는 자율적으로 상호작용(=실제 인간과 유사한 개인화 행동으로 볼 수 있음)하면서 실제 interaction record와 유사한 일관된 의사결정을 수행하는지를 탐색하도록하고, <br> (2) 이후 상호작용으로부터 얻은 피드백을 바탕으로 사용자와 아이템 에이전트가 서로의 메모리를 반영하고 수정할 수 있도록하는 collaborative reflection mechanism을 설계하여 <br> 이를 통해 사용자와 아이템 선호가 상호적으로 축적되고 이후 상호작용에도 반영되도록한다. 
= 상호작용을 통해 CF 아이디어 모델링<br>
<br>
평가는 실제 데이터셋 기반 실험 수행 후 평가를 했으며 그 결과 user-item interaction을 효과적으로 시뮬레이션 했으며, 기존 추천 모델과 LLM 기반의 추천 방법들에 비해 더 우수한 성능을 보였다고한다. (마지막절에서 추가 설명O )


## 2. Methodology
**AgentCF는 user 에이전트와 item 에이전트를 협업학습하는 방법**

### 2.1 Preliminaries
#### 전통적인 CF 설정
사용자 U = {u}, 아이템 I= {i}, 상호작용 기록 D = {<u,i>}, u와 i에 대한 선호도함수 f(u,i)가 있다고 할 때, f(u,i)는 positive일 경우 i+, negative일 경우 i- .<br>
ex.<br>BPR : 2개 아이템의 선호도를 비교하는 personalized pairwise ranking 모델 구축 <br>
NCF : 신경망 사용하여 user item 관계 학습<br>
이후 f(u,i)의 학습은 ML의 표준적인 함수 피팅 문제(?) 로 변환되며 학습데이터 D{u,i}에 기반한다. (기울기 기반 function fitting 문제)

#### AgentCF의 설정
(사진추가)
**AgentCF의 전체 프레임워크**
1. 사용자 에이전트와 아이템 에이전트가 먼저 자율적으로 상호작용한다. <br>
2. 이후 자신의 의사결정과 실제 데이터간 차이를 반영해서 메모리를 수정한다. <br>
3. 이 과정에서 사용자와 아이템의 시뮬레이션된 신호가 점점 집계(aggregate)될 것이고, 다른 에이전트에 전파(propagation)될 것이다. <br>    

<br><br>
AgentCF는 전통적인 추천 기법의 설정을 따르면서 LLM 기반의 에이전트를 결합하도록 한다. <br>
 <span style="background-color: #fff3cd">사용자나 태스크 중심의 에이전트만 고려하던 기존 방식들과 달리, item agent도 고려하고자 하는 것이 핵심이다. </span>
 - user agent : 실제 사용자 선호 모델링
 - item agent : 아이템의 특성과 잠재적 사용자 선호도 반영

이 설정에서는 user, item 간 원래 상호작용이 두 에이전트간 자율적 상호작용으로 시뮬레이션 됨 <br> 그리고 랭킹 문제(LLM 기반 함수로 사용자에게 후보 아이템 리스트의 순위를 매기는..)에 집중한다. 이때 LLM은 고정 !!!

>  <span style="background-color: #fff3cd"> 메모리를 가진 LLM</span>이라고 하면 깔끔한 답이다.

> **Q. 왜 LLM을 고정시키지? 안 고정 시키면 어떻게 되는가?** <br> 
> A. 본 논문은 LLM은 고정시키고 메모리만 업데이트하며 학습시키는데 만약 LLM을 고정시키지 않으면 어떤 행동이 일어났을 때 메모리 때문인지 LLM 파라미터? 때문인지 구분 불가일듯 (해석이 어려워질듯) <br>

Agent CF 구현하기 위해서는 두개의 매커니즘을 활용한다. <br>
- 메모리 : 과거 상태, 행동, 맥락 저장
- reflection : 에이전트의 상태나 인식 수정    
<br><br>
이러한 방식은 기존에도 연구되었으나 대부분 태스크 중심이거나 사용자 중심 시뮬레이션에 초점이 맞춰졌었다. <br>
<span style="background-color: #fff3cd">아까도 말했듯이 이 논문은 item이 추가됨에 의의가 있는거임!!</span>
<br>
아이템 에이전트가 없으면 CF의 핵심 개념인 like alike(비슷한 사용자가 좋아한 아이템 추천)을 제대로 표현하기 어려움. <br>

#### collaborative memory based optimization
단순히 사용자의 과거 데이터를 LLM에 넣는게 아니라, 사용자 에이전트와 아이템 에이전트의 상호작용을 기반으로 함께 개선을 시켜나간다고 한다... <br>각 단계에서 에이전트가 자율적으로 상호작용하고 실제 데이터와 차이를 reflect 하여 메모리를 업데이트하면서 실제 사용자의 행동패턴이 더 잘 반영되게 된다고 한다. 

#### implicit preference propagation
기존 연구들과 달리 각 상호작용마다 user, item  <span style="background-color: #fff3cd">둘다 메모리를 업데이트해서 특정 사용자의 선호가 다른 유저에게도 전파되도록</span> 한다.


### 2.2 Collaborative Agent Optimization
유저와 아이템 에이전트에 메모리 모듈을 부여해서 각각의 고유한 특성과 협업 정보를 유지하도록하는데 <br> 메모리 디자인을 어떻게 하는지?
#### User Memory
사용자의 선호 정보를 반영하는 다양한 정보들을 저장하는 역할을 한다. <br>
현실에서 사용자 선호는 시간에 따라 변하기 때문에
- 단기 메모리 $M_u^s$ : 자연어 형태, 최근 업데이트된 선호 표현
- 장기 메모리 $M_u^l$ : 과거 선호 기록, 사용자 선호가 시간에 따라 어떻게 변화했는지
<br><br>
새로운 상호작용 발생하면 $M_u^s$에서 직접 사용, $M_u^l$에서 관련 정보 검색
<br>

#### Item Memory
- 아이템 자체의 특성 + 해당 아이템 선호하는 사용자들의 특징 <br>
- 사용자 메모리에 비해 안정적이므로 $M_i$만 사용한다. <br>
- 초기에는 제목, 카테고리 등으로 초기화 -> 사용자와 상호작용 통해 사용자 선호벙보 지속적으로 업데이트
- 전파 가능하게함 + 실제 추천시스템에서 아이템 특성 풍부하게 함
  
#### Autonomous Interactions
초기화된 에이전트가 주어졌을 때 목표 : 이들을 최적화하여 실제 user item 상호작용을 시뮬레이션 하도록. <br><br>
이를 위해 시뮬레이션된 에이전트와 실제 사용자만 행동 일치여부 분석<br>-> agent 업데이트 위한 피드백 제공 

BPR에서 두 아이템간 선호 비교하는 foward computation 과정으로도 볼 수 O <br>
구체적으로, 실제 사용자 행동 시퀀스 = 학습데이터 (각 상호작용 단계에서 사용자 에이전트가 선택할 후보로써 positive i+, negative i- 대조쌍 제시) <br>
<br>
개인화 부족한 에이전트는 주로 상식적 지식에 의존하여 인기있는 아이템 / 리스트 상단에 위치한 아이템 선택하는 경향O (discrimination difficulty 높이기 위해) <br>
-> 인기 bias, 위치 bias 도입 -> 데이터셋에서 인기 높은 negative item 샘플링한 뒤, 이를 positive item 보다 앞에 배치함<br>
이후 u는 후보들 중 하나의 item i0을 선택한 후 그 선택에 대한 설명 $y_exp$를 생성한다. (에이전트는 자신의 메모리에 저장된 선호정보 + 각 후보 아이템의 메모리 정보 함께 활용)<br>
<span style="color: red">positive/negative -> LLM이 선택 = forward 단계 </span> <br>
(BPR의 둘 중 뭐가 좋아?를 LLM으로 구현한 것)

#### Collaborative Reflection and Memory Update
앞선 과정들을 에이전트가 상호작용을 시뮬레이션 할 수 있게 하는데, 이제 <span style="color: red">에이전트의 선택을 실제 상호작용 데이터와 비교해서 피드백 신호를 생성하고, 이를 활용해서 user agent, item agent를 협업적으로 최적화해야한다. (=backward update) </span><br><br><br>
<span style="background-color: #fff3cd">이 과정에서 LLM은 고정이고 기존 추천 모델들처럼 명시적인 gradient를 학습과정에 쓰지 않고 대신 메모리 업데이트를 한다!</span> <br>
에이전트가 선택한 아이템 i와 생성한 설명 y_exp를 바탕으로 user와 item 에이전트 모두에게 자신의 시뮬레이션된 선호와 특성에서의 misconception 수정하도록함 = collaborative reflection<br>

![AgentCF(1)](/assets/images/AgentCF(1).png) 
![AgentCF(2)](/assets/images/AgentCF(2).png)

task 중심의 reflection, user 중심의 reflection 모두 기존에는 self reflection으로 이루어졌었는데 본 방식은 user, item 양쪽 동시에 수행한다는 점에서 차이가 있다. <br> <br>  
업데이트 방식 : <br> 
정답(실제 데이터와 일치) 선택 : 정답 알려주고 관련 정보 메모리에 저장<br> 
오답 선택 : Collaborative Reflection 통해 메모리 수정<br> <br> 
(단기메모리를 장기메모리에 추가하여 추루 상호작용에서 다시 활용하며, 기울기 대신 reflection으로 메모리를 업데이트한다 = AgentCF의 backward)<br> <br>
